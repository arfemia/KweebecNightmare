# npc/ - the Grove Warden guide spawner

Router for `npc/`. Kweebec's thin policy layer over the lifted `ziggfreed-common` NPC primitives ([`com.ziggfreed.common.npc`](../../../../../../../ziggfreed-common/src/main/java/com/ziggfreed/common/npc/CLAUDE.md): `NpcSpawnService`, `ActionOpenDialogue`/`ZigOpenDialogue`, `NpcDialogueDepsRegistry`, `NpcActions`). The spawn primitive + press-F action live in common; this package keeps only the kweebec-specific placement policy, exactly as the MMO's `QuestGiverSpawnService` layers on its primitive.

- **[`KweebecGuideSpawn`](KweebecGuideSpawn.java)** keeps EXACTLY one guide per eligible world by **reconciling against the LIVE world entity store** on `PlayerReadyEvent` (registered in [`KweebecNightmarePlugin.setup`](../KweebecNightmarePlugin.java)), inside `world.execute`. `reconcileGuide` runs ONCE per world per boot (an in-memory `reconciledThisBoot` set; PlayerReady world tasks are serialized on the world thread): it scans for guide NPCs via `store.forEachChunk(NPCEntity.getComponentType(), ...)` matching `NPCEntity.getRoleName()`, then `findGuides`-> if >=1 it KEEPS the first (re-records its UUID) and `store.removeEntity`s the rest, else it places a fresh one near the world spawn via `NpcSpawnService.resolveSpawnPosition` + the configured offset. **The world is the source of truth, NOT the file marker** - the marker is only a fast per-boot hint + the UUID record. The PRIOR marker-only gate (`hasSpawned`->`markSpawned`) was sound going FORWARD (serial task queue + synchronous `store.addEntity`, verified in the 0.5.3 decompile), but a proxy can't SEE the world: it could neither clean up duplicates an older build had already SAVED into the world (the "one guide stacked per restart" bug, since the guide NPC persists natively) nor recover a marker/world drift. The scan handles both, and is correct whether or not the guide persists across restarts. A failed scan/place drops the per-boot claim so the next join retries.
- **[`KweebecGuidePlacementStore`](KweebecGuidePlacementStore.java)** - the kweebec mirror of the MMO's `MmoNpcPlacementStore` marker, slimmed to a persisted `spawnedWorlds` set + a per-world guide UUID at `<data dir>/guide-placements.json`. **Now a hint/UUID record, no longer the dedup authority** (the live-world scan in `KweebecGuideSpawn` is). `init(dataDir)` (called in `setup` before the player-ready hook) loads it; `markSpawned`/`recordGuide` save it. The recorded UUID lets `reposition` (the `/kweebec spawnguide` debug hatch) despawn the old guide - even one from a prior session - before placing a fresh one. World-thread only; one small JSON write per world per boot.
- **Config-driven via [`KweebecGuideConfig`](KweebecGuideConfig.java)** (`<data dir>/guide.json`, defaults written on first run) - the kweebec mirror of the MMO's `spawn-hub.json` / `SpawnHubConfig`: `enabled`, `role`, a `worlds` allowlist (default `["default"]` so ONLY the main overworld gets the guide; `"*"` = every world, empty = none), `offset` {x,y,z}, and `yaw`. **`PlayerReadyEvent` DOES fire for instance worlds** (the kweebec round instance + the creative hub - the old "no PlayerReadyEvent in instances" assumption was wrong), so they are kept guide-free by simply not being in the `worlds` list, NOT by a special-case check.
- **Press-F wiring is in the role asset, not here.** `Server/NPC/Roles/KweebecNightmare/KweebecNightmare_Guide.json` (a `Type: Generic` Passive role copied from the MMO's `MMO_QuestGiver`) carries the `InteractionInstruction` that fires `{ "Type": "ZigOpenDialogue", "Dialogue": "nightmares_intro", "ContextNpc": "guide" }`. The action + its `ZigOpenDialogue` registration + the `DialoguePageDeps` provider all live in common / [`../dialogue/KweebecDialogue`](../dialogue/KweebecDialogue.java); the plugin calls `NpcActions.register()` + `NpcDialogueDepsRegistry.set(KweebecDialogue::deps)` in `setup()` BEFORE the role asset loads, or `{ "Type": "ZigOpenDialogue" }` fails to parse.

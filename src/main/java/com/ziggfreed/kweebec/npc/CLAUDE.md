# npc/ - the Grove Warden guide spawner

Router for `npc/`. Kweebec's thin policy layer over the lifted `ziggfreed-common` NPC primitives ([`com.ziggfreed.common.npc`](../../../../../../../ziggfreed-common/src/main/java/com/ziggfreed/common/npc/CLAUDE.md): `NpcSpawnService`, `ActionOpenDialogue`/`ZigOpenDialogue`, `NpcDialogueDepsRegistry`, `NpcActions`). The spawn primitive + press-F action live in common; this package keeps only the kweebec-specific placement policy, exactly as the MMO's `QuestGiverSpawnService` layers on its primitive.

- **[`KweebecGuideSpawn`](KweebecGuideSpawn.java)** auto-spawns the guide role once per world on `PlayerReadyEvent` (registered in [`KweebecNightmarePlugin.setup`](../KweebecNightmarePlugin.java)), near the world spawn via `NpcSpawnService.resolveSpawnPosition` + the configured offset, all inside `world.execute`. Idempotency is the **persistent, kweebec-owned** [`KweebecGuidePlacementStore`](KweebecGuidePlacementStore.java) - it checks `hasSpawned(world)` then `markSpawned` only AFTER a successful place (mirrors hyMMO's `maybeAutoSpawnHub`). **It MUST be file-backed, not an in-memory `Set`: the guide NPC persists natively in the world's entity store, so a transient marker resets every boot and stacks a fresh guide beside the saved one each restart** (the shipped bug; "still spawning on every connect"). It is kweebec-owned (its own `guide-placements.json`), NOT a lifted singleton placement store - the reason that store never lifted into common was cross-mod clobbering, NOT that it should be transient.
- **[`KweebecGuidePlacementStore`](KweebecGuidePlacementStore.java)** - the kweebec mirror of the MMO's `MmoNpcPlacementStore` once-per-world marker, slimmed: a persisted `spawnedWorlds` set + a per-world guide UUID, at `<data dir>/guide-placements.json`. `init(dataDir)` (called in `setup` before the player-ready hook) loads it; `markSpawned`/`recordGuide` save it. The recorded UUID lets `reposition` (the `/kweebec spawnguide` debug hatch) despawn the old guide - even one from a prior session - before placing a fresh one. World-thread only; one small JSON write per world per boot.
- **Config-driven via [`KweebecGuideConfig`](KweebecGuideConfig.java)** (`<data dir>/guide.json`, defaults written on first run) - the kweebec mirror of the MMO's `spawn-hub.json` / `SpawnHubConfig`: `enabled`, `role`, a `worlds` allowlist (default `["default"]` so ONLY the main overworld gets the guide; `"*"` = every world, empty = none), `offset` {x,y,z}, and `yaw`. **`PlayerReadyEvent` DOES fire for instance worlds** (the kweebec round instance + the creative hub - the old "no PlayerReadyEvent in instances" assumption was wrong), so they are kept guide-free by simply not being in the `worlds` list, NOT by a special-case check.
- **Press-F wiring is in the role asset, not here.** `Server/NPC/Roles/KweebecNightmare/KweebecNightmare_Guide.json` (a `Type: Generic` Passive role copied from the MMO's `MMO_QuestGiver`) carries the `InteractionInstruction` that fires `{ "Type": "ZigOpenDialogue", "Dialogue": "nightmares_intro", "ContextNpc": "guide" }`. The action + its `ZigOpenDialogue` registration + the `DialoguePageDeps` provider all live in common / [`../dialogue/KweebecDialogue`](../dialogue/KweebecDialogue.java); the plugin calls `NpcActions.register()` + `NpcDialogueDepsRegistry.set(KweebecDialogue::deps)` in `setup()` BEFORE the role asset loads, or `{ "Type": "ZigOpenDialogue" }` fails to parse.

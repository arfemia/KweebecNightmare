# event/ - outbound native events + the unified damage observer

Router for `event/`. The mod-root `CLAUDE.md` "MMO Skill Tree integration" section is the wider context. This package holds the outbound-event integration seam (zero MMO dependency) AND the one damage-observer system.

- **[`RoundEvents`](RoundEvents.java) fires the `api/` `IEvent<Void>` POJOs** (`RoundStarted`/`RoundCompleted`/`KweebecRoundScored`/`PlayerCocooned`/`PlayerRescued`) on the shared Hytale bus: resolve the dispatcher, guard on `hasListener()` (silent no-op with zero listeners), dispatch synchronously on the calling thread, whole body try-guarded. Fire from a WORLD-thread context so a listener can resolve a player then hop via `world.execute`. `KweebecRoundScoredEvent` (carrying the per-player `PlayerScore` map) fires AFTER `RoundCompletedEvent` from `RoundService.resolve` - the MMO capstone reward hook.
- **A consumer (MMO) subscribes via the reflective `registerGlobal` adapter** - it loads these event classes from THIS mod's classloader so `Class` identity matches across the jar boundary (the `AnglersAlmanacAdapter` pattern). When MMO is absent the mod self-contains its own rewards. The event POJOs live in the separate `api/` Gradle module (bundled into the jar minus its `META-INF/services` so the engine does not double-register); never bundle the MMO api jar back the other way.
- **[`KweebecDamageSystem`](KweebecDamageSystem.java) is the ONE damage observer** (a `DamageEventSystem` in `DamageModule.get().getFilterDamageGroup()`, registered in `KweebecNightmarePlugin.setup` like `CocoonOnDeathSystem`; `getQuery()` = `Query.any()`). It never initiates damage. Two jobs: (1) **throw-to-stun** - a Moonbloom-caused hit on a non-player mob ANYWHERE freezes it via Perfect Utils `StunMobAPI` + the vanilla `Stun` visual effect (an `isStunned` guard makes it fire once per window; players in the splash are skipped, so no friendly-stun); (2) **damage-taken scoring** - real damage to an in-round survivor accumulates onto `PlayerRoundState.damageTaken`.
- **GOTCHA - how to identify a thrown item's hit server-side (a reusable Hytale technique).** A burst/AOE interaction's damage (`DamageEntityInteraction`) is sourced from `context.getOwningEntity()` = the **SHOOTER** (the player), NOT the projectile entity (`DamageEntityInteraction.java:192` `new Damage.EntitySource(ref)`), and it is an `EntitySource`, never a `ProjectileSource`. So you CANNOT read the projectile config off the damage source, and you cannot tell a thrown-item hit from a melee hit by the source alone. The robust identifier is a **custom `DamageCause`**: author `Server/Entity/Damage/<Id>.json` (`{Parent:Physical, Inherits:Physical}`, a pack-authorable asset type), make the burst's `DamageCalculator.BaseDamage` key that cause id, and match `damage.getDamageCauseIndex()` against `DamageCause.getAssetMap().getIndex("<Id>")` (cache only a positive index - it resolves after this class loads). The cause travels with the damage independent of source/world; the thrower for attribution is still `EntitySource.getRef()` (the owning shooter).

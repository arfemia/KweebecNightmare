# i18n/ - localization key registry + the .lang prefix contract

Router for `i18n/`. The mod-root `CLAUDE.md` Gotchas section has the one-line rule; this is the detail. All player-facing text resolves through a key constant here - no raw display strings, from day 1.

- **[`Lang`](Lang.java) is the key registry + `Message` factory.** Every constant is the FULL final key (e.g. `kweebecnightmare.hud.timer`); `Lang.msg(key)` wraps `Message.translation(key)` (resolved client-side). Asset files reference the same full keys (`instance.bson` `TitleKey`/`SubtitleKey`, the Blight role `NameTranslationKey`, `KweebecMode.nameKey`).
- **`.lang` FILE keys are NOT prefixed - the FILENAME supplies the namespace.** The engine (`I18nModule.getPrefix` -> `loadMessagesFrom`) derives the prefix from the file name `kweebecnightmare.lang` -> `kweebecnightmare` and prepends `<prefix>.` to EVERY key in the file. So the file holds `hud.timer = Time {0}` and the engine builds the final key `kweebecnightmare.hud.timer` (the form `Lang` references). Writing the full `kweebecnightmare.hud.timer` IN the file doubles it (`kweebecnightmare.kweebecnightmare.hud.timer`) and every HUD/title/toast string falls through to the raw key on screen (the 0.2.0 bug). Mirror the parent mod's `mmoskilltree.lang` (unprefixed keys). `#` starts a comment line.
- **en-US is the only shipped locale** (`Server/Languages/en-US/kweebecnightmare.lang`), hand-authored and copied verbatim by `processResources` (no `generateEnglishLang`-style generator, unlike the parent mod). Adding a string = add the `Lang` constant (full key) AND the matching unprefixed line in the `.lang`. A new player-facing string is always a new key, never a literal.
- **RICH TEXT (colour/bold/italic) = native `<color is="#hex">`/`<b>`/`<i>`/`\n` markup in the `.lang` value, rendered on a Label's `TextSpans` - NEVER `.Text`** (which shows the tags literally - the original guide-dialogue bug). The client parses it, per-locale. The guide dialogue node text (`dialogue.nightmares_intro.greet.text` etc.) does exactly this; the page sets `#NodeText.TextSpans`. `TextButton` labels (the dialogue option rows) have NO `TextSpans`, so they take no markup - they are colour-coded by their `DialogueOptionStyle` button style instead. The interaction Hint uses the `[{key}]` keybind placeholder (`Press [{key}] to ...`), like vanilla `server.interactionHints.*`. Full authority: `../../../../../../.claude/research/hytale-rich-text-textspans.md`.

# arena/ - the Chase grove skeleton (anchors + prefab stamping)

Router for `arena/`. The mod-root `CLAUDE.md` (loads with the kweebec subtree) has the Path-A-instance overview; the kweebec techspec (linked from the mod root) has the verified `PrefabUtil.paste` signature. Gameplay runs ENTIRELY off the anchors here, so a round is playable before any prefab pastes - the geometry is a visual layer, never a logic dependency.

- **[`ArenaLayout`](ArenaLayout.java) is the fixed coordinate authority** - `SPAWN`/`GATE`/`ESCAPE`/`HUNTER_DEN` + the cave anchors (`caveShrineAnchors(count, seed)` a seed-shuffled subset at `CAVE_STAND_Y`, used to CARVE the cave shafts) + (0.4.x) `SURFACE_WORLDGEN_SHRINES` (=3, the win-count term AND the surface TOP-UP target) and `WORLDGEN_SHRINE_XZ` (now only a FALLBACK Moonbloom-cluster spot used before detection runs; the REAL positions come from [`ShrinePlacement`](ShrinePlacement.java) and live on `ChaseState.surfaceShrinePositions()`). **The surface shrine RING is GONE as the placement source, but `shrineAnchors(count, seed)` is REVIVED** as `ShrinePlacement`'s spaced TOP-UP candidate ring (no longer vestigial). Surface shrines are WORLDGEN-SCATTERED host prefabs: a single `Mesh2D` `Type:Prefab` biome prop weighting the 3 `Shrine_*` hosts (each a copied vanilla crypt/ruin with the furnace baked as a 4-cell multiblock), discovered via the furnace interaction. **The trap (now in the `worldgen-v2` skill):** a worldgen prop path must resolve to a FOLDER (`Shrine_*/Shrine_*.prefab.json`, not a bare file - else it silently loads nothing). NOTE: a `List` fixed-position prop DOES work (the prop pass is world-positioned); the surface shrines use `Mesh2D` scatter by choice (random), but a `List` prop of fixed `{X,Y,Z}` (folder paths, Y in `[0,320]`) is the deterministic exact-count alternative if a fixed layout is wanted. `RuleSet.shrineCount` no longer drives placement. Proximity gates use `INTERACT_RADIUS_SQ` (rescue) and `ESCAPE_RADIUS_SQ`.
- **[`ShrinePlacement`](ShrinePlacement.java) reconciles the SURFACE shrine count (detect + top-up).** Because the surface hosts are RANDOM `Mesh2D` scatter (no count/spacing guarantee), at round start it (1) FORCE-LOADS the play core (`DETECT_RADIUS` r70) via `World.getChunkAsync` - the Chase instance has `IsUnloadingChunks:true`, and an ECS query sees only LOADED chunks, so this is mandatory; (2) runs ONE `store.forEachChunk(Query.and(FromWorldGen, PersistentDisplayName))` filtered to the `kn_shrine_marker` raw-text SENTINEL - each host prefab bakes ONE hidden marker entity at its anchor (`LoadEntities:true`, so worldgen tags it `FromWorldGen`); (3) tops up ONLY the deficit to `SURFACE_WORLDGEN_SHRINES` at spaced `shrineAnchors(count, seed)` anchors cleared by `ExclusionMask.shrinePlacementMask()` + a min-distance to detected; (4) publishes the resolved positions to `ChaseState` and plants the initial Moonbloom there. **Degrade-to-deterministic**: if detection finds nothing it places the full target spaced, so the round is always winnable. Block-scan is the pocket fallback (force-load is shared) if the baked markers ever prove unreliable in-game. The marker sentinel string MUST stay in sync with the 5 `Shrine_*` prefabs' `PersistentDisplayName.RawText`.
- **GOTCHA - the BIG hosts (large crypt / dungeon) almost never reach a round; small crypts always do.** `ShrinePlacement` only force-loads + scans the r70 `DETECT_RADIUS` core and tops up the deficit with `TOPUP_HOST_PREFAB` = `Shrine_Crypt_01` (SMALL crypt) ONLY. `Shrine_Crypt_Large` (Scale 80 / SkipChance 0.65) and `Shrine_Dungeon_Underground_01` (Scale 90 / SkipChance 0.7) are sparse WHOLE-GROVE scatter, so for a given deterministic seed their expected count inside r70 is < 1 (often 0) and they are never substituted in the top-up. Symptom: "the large crypt / dungeon don't spawn." It is NOT the houses toggle (the houses prop was already `Skip:true` when this system landed, and the large crypt did not exist in the runtime-`ArenaBuilder` era the bug report remembers). Fix: top up with one large crypt + one dungeon (or place them deterministically), not whole-world scatter + small-crypt-only top-up. Full trace + secondary findings: [[kweebec-worldgen-shrine-not-in-core]].
- **0.3.0 - corrupted village ruins + per-round variety.** The per-round world auto-seeds (the `instance.bson` `Seed` was removed; read back into `RoundInstance.worldSeed`), seeding the ring rotation, cave subset, AND the structure selection from ONE coherent seed. [`StructureCatalog`](StructureCatalog.java) (kweebec footprint policy + role mapping) delegates the seeded selection to ziggfreed-common's `WeightedPrefabPlacementConfig.select(...)` (the migrated-to-common placement table, content at `Server/ZiggfreedCommon/PrefabPlacements/`); [`ExclusionMask`](ExclusionMask.java) (the shared gameplay keep-clear discs: spawn courtyard, shrine ring, gate corridor, den, cave anchors) keeps ruins off the beats. `ArenaBuilder.pasteStructures` (seed = `round.worldSeed()`) pastes 3-5 surface-snapped `Corrupted_*` ruins (cover/chokepoint/beacon/landmark), and the centerpiece is now `Corrupted_Well`. Ambient ruins also scatter gen-time via the biome `Props[1]` `Type:Prefab` prop (sparse, `LoadEntities:false`).
- **The arena is NATURAL ROLLING grove terrain (1C: the flat play disc is GONE), bounded by the finite EDGE CLIFF.** The Chase instance worldgen is the pack-authored `KweebecNightmare_Grove` (WorldStructure `Base`=80) whose biome density now rolls ~y72-88 across the WHOLE footprint, then a sheer cliff into the void at ~r112 (the design keeps the edge cliff). A SMALL r6 origin **spawn clearing** (a `Cylinder` gating the hill noise to 0 inside r6) keeps spawn flat + deterministic at y80 so `instance.bson` `SpawnProvider.SpawnPoint.Y`=80 stays valid - a courtyard, NOT the old disc (the original Y=65 buried players, the 0.2.0 bug). Off-center anchors are NOT flat-guaranteed, so their pastes FLOOR-SNAP to the real surface (see the ArenaBuilder bullet).
- **The `KweebecNightmare_Grove` biome density is ALL-ADDITIVE + fail-safe**: `Sum(wideRamp, hillNoise, Multiplier(caveMasks,-8), Multiplier(edgeMask,-100000))`. The two r70 flat-core `Cylinder` masks were REMOVED (1C) so hills + caves apply EVERYWHERE; only the r112 edge-cliff Cylinder remains. No top-level Min/Max, so any optional term degrading to `ConstantValueDensity(0.0)` just drops that feature - the ground NEVER airs out. Hills = a WIDE ramp curve (crossing y=80) summed with `Normalizer` noise (+-0.20, ~+-8 blocks), the noise wrapped in a `Multiplier` with the r6 spawn-clearing `Cylinder`; NOT the sharp 1-block curve (which floods solid-to-sky when summed). Caves carve only below ~Y66 (a fixed buffer under the lowest valley). See [[kweebec-worldgen-v2-deep-pass]] + the biome JSON `$Comment` hand-trace.
- **The biome scatters ALL ambient decoration via `Props[]`** (1C): ground cover (`Mesh2D` -> `Column` props floor-snapped by a `ColumnLinear` Scanner: dead grass/tall grass, dead bush / twisted bramble / glowing purple mushroom) AND the ambient blight **TREES** (a `Type:"Prefab"` prop over native `Trees/Poisoned`/`Ash_twisted`/`Petrified_Dead`/`Dry_Dead`/`Ash_Dead`, weighted + `SkipChance`-thinned, mirroring the proven vanilla `Example_Prop` shape). **CORRECTION (decompile-verified):** these props specify NO `BlockMask`, so the engine uses the DEFAULT empty mask = CARVE-ALL (`BlockMask.canReplace` returns true for everything); the old "`BlockMask.DontReplace` so trunks don't gouge soil" claim is FALSE. Trunks/ruins avoid soil because their `Floor` directionality PATTERN gates placement (Empty over `Soil_Grass_Dry`), not a mask. See [[kweebec-worldgen-shrine-not-in-core]]. The old runtime `ArenaBuilder` foliage ring is GONE (the design's "trees from worldgen, not manual paste"; it also cost a world-thread paste stall + the worldgen race). In the biome JSON `$NodeEditorMetadata`/`$Links` are dead node-editor metadata. The biome JSON is NOT gradle-compiled - validate terrain/scatter in-game. **FULL ATMOSPHERIC PASS** (Seam 1 of [[kweebec-worldgen-both-seams]]): the grove biome now has SIX `Props[]` layers - `KN_Trees` (widened dead/ash/petrified/burnt palette + rare Ash_twisted Large/Giant landmarks), `KN_Ruins` (pack `Corrupted_*`), `KN_Grass`, `KN_Brush` (bush/bramble/mushroom), plus the new `KN_Debris` (fallen logs / stumps / petrified logs / mossy boulders, Scale 14) and `KN_Graves` (Cursed_Crypt tombstones + sign-post, Scale 40, rare). All new layers are CLONED from the proven `Corrupted_*` Prefab-prop shape (a malformed Prefab prop airs out the WHOLE biome) and every prefab path was verified to resolve. The big `Crypt_Entrance` set-piece is deliberately NOT worldgen-scattered (it carves a deep pit; a random drop onto a shrine/spawn would break the chase) - it belongs on the runtime, `ExclusionMask`-protected seam.
- **[`ArenaBuilder.build`](ArenaBuilder.java) stamps the RUNTIME beats**: the exit, the CAVE shafts (force-carved), the centerpiece + corrupted ruins; the gate is deferred. **Surface shrines are NO LONGER stamped here** - they are WORLDGEN-SCATTERED host prefabs (a `Mesh2D` `Type:Prefab` biome prop over the `Shrine_*` folders), so `placeShrineBlock`/`setShrineFurnace` are GONE (a single `setBlock` of the 2x2 furnace rendered wrong; the furnace is baked into the prefab instead). Best-effort: blocking prefab I/O on a background thread, then each paste hops to the world thread (a missing prefab is logged + skipped). **Each paste FLOOR-SNAPS to the real surface**: `paste(...)` probes the local top-solid Y with `ziggfreed-common`'s `SurfaceProbe` inside `world.execute` (degrading to `at.y-1` on a miss). **The probe SKIPS surface decoration** (`SURFACE_DECORATION_LISTS` = `TreeWoodAndLeaves` + `AllScatter` via `BlockTypeLists.keys(...)`) so a beat lands on ground, not a tree/leaf. **CORRECTION to the old both-seams claim**: objectives CAN ride worldgen when expressed as a registered interactable BLOCK (the furnace) baked into a prefab + discovered via its interaction (no position recompute / placement registry needed) - see [[kweebec-worldgen-both-seams]] + the `hytale-modding` `worldgen-v2.md` objective-as-block recipe (reusable for the MMO). [`Anchor`](Anchor.java) is the `(x,y,z,yaw)` record + `horizontalDistanceSq`.
- **`ArenaBuilder` cluster planters (harvestable glow-throwable scatter, 0.5.0).** `plantClusters(round, world, prefabKey, perShrine, scatter, salt)` is the ONE generalized core: a ring at each resolved surface shrine + a seed-deterministic grove scatter (salted by prefab so two throwables do not stack on a tile), each a one-block plant-cluster prefab floor-snapped via `paste`. `plantMoonbloom` now just delegates to it. `plantClusterRing(world, prefabKey, cx, cz, radius, count[, angleOffset])` rings a POINT - the boss-phase Emberbloom placer `BossController` calls around the Warden; the `angleOffset` overload rotates the ring per respawn wave (golden angle) so a stationary boss's successive Emberbloom rings land on FRESH tiles instead of stacking atop each other (BossController scatters the board supply ONCE per phase, then re-rings the close supply only). `surfaceDecorationKeys()` exposes the `SURFACE_DECORATION_LISTS` foliage-skip set as the ONE authority any runtime spawn shares (the boss + its adds snap PAST the canopy through it, like the hunter, so the Warden never rises on top of a tree). `plantGroveThrowables(round, world, salt, wavesOnly)` iterates every enabled [`../asset/GroveThrowableConfig`](../asset/CLAUDE.md)`.placeable()` entry (Gust/Mire + any pack variant; shipped disabled), gated by `MinCorruptionTier`, driven for the INITIAL supply from `ShrinePlacement` + each Moonbloom respawn wave from `ChaseMode`. All async + best-effort like the other pastes.
- **Underground shrines are a tunable feature** (`RuleSet.caveShrineCount`, default 2): `ArenaBuilder.pasteObjectives` carves that many cave shafts directly from `ArenaLayout.caveShrineAnchors(count, seed)` (deterministic anchors SPREAD to r~38 across the four quadrants, capped at 4; `ExclusionMask` auto-tracks them) - the cave count is part of `ChaseState.totalShrines`, so `allShrinesLit()` REQUIRES descending into every cave + returning. `pasteCaveShaft` FLOOR-SNAPS each shaft top to the LOCAL surface (`SurfaceProbe`) and force-carves it (`force=true` -> `PrefabUtil` `chunk.setBlock`, UNCONDITIONAL, writes `Empty` so the prefab CARVES the rock). **The shrine FURNACE is BAKED into the shaft prefab at the chamber** (the cyan beam + the runtime `setShrineFurnace` are GONE), discovered via its interaction like the surface shrines. Idempotent re-carve: the resolved surface Y is remembered per-cave in `ChaseState.caveCarveY` so a +4s/+9s re-paste reuses it instead of re-probing (which would stack a second shaft). Cave style: `SHAFT_PREFABS` indexes the runtime force-carved descents - currently `[0]` `Relight_Shaft` (a facing-independent spiral staircase) only. **The multi-level underground DUNGEON (`Shrine_Dungeon_Underground_01`) was MOVED off this runtime seam to WORLDGEN scatter** (biome `KN_ShrineDungeon` prop): a `Type:Prefab` worldgen prop writes EVERY stored cell incl. `Empty` (`PrefabProp.place`), so its authored `Empty` descent shaft CARVES out of the rock AT GENERATION TIME - the same carve the runtime seam did, now gen-time + reliable (no worldgen race, no force-paste). Its anchor IS the surface entrance (`anchorY 36`, body descends ~36 blocks to the baked furnace); it carries the `kn_shrine_marker` so `ShrinePlacement` detects it as a host. Surface (worldgen) beats are `force=false`.
- **There is NO separate Heartwood Gate PREFAB any more** (the old purple/blue `Gate` light archway was removed). `ChaseMode.openGate` fires the gate-open beat (titles / exit marker / hunter alert) AND reveals the exit platform via [`ArenaBuilder.pasteExit`](ArenaBuilder.java) - the exit is NOT pasted at round start, only at this climactic beat. `ArenaBuilder.pasteGate` + `GATE_PREFAB` are gone. The win is the CO-OP EXTRACTION HOLD (`ChaseMode.checkExtraction`: the required group holds the platform within `ESCAPE_RADIUS_SQ` of `ESCAPE`); the gate-cross instant win (`crossing GATE.z`) was REMOVED with the rework. `ArenaLayout.GATE` stays (the corridor anchor + hunter logic + `bossBarsGate` concept).
- **The exit platform is the `Extraction_Pad` prefab** (`EXIT_PREFAB`, pasted by `pasteExit` at `ArenaLayout.ESCAPE` AT GATE-OPEN, floor-snapped - NOT in `pasteObjectives`, which now only carves the cave shafts): a vanilla **Frost Elemental Circle** medallion copied into the pack by [`tools/build_extraction_pad.py`](../../../../../../tools/build_extraction_pad.py) (vanilla prefab keys do NOT resolve at runtime, so it is committed into `Server/Prefabs/KweebecNightmare/`, like `Corrupted_Well`). The tool keeps the SOLID blocks (drops `Empty`/`Block_Spawner_Block`/the construction signs/the water `fluids`) - the FULL elemental circle incl. its central crystal medallion is KEPT, NO hand-built portal - and crowns the centre with the mod's own glowing **`KweebecNightmare_Moonbloom_Plant`** as the exit marker (`moonbloom_cluster`: one plant on top of each central full-face column; the Moonbloom plant needs `Support.Down=Full`, so it is placed ONLY on full blocks - not on half/stairs/crystal-model tops - or the `force=false` paste would skip it). **DO NOT add lookalike portal blocks**: an earlier attempt used `Build_Lightsource_Purple` (does NOT exist -> magenta missing-texture); valid lightsource colours are Blue/Cyan/Pink/Red/White/Yellow, no purple, and a real portal-surface block teleports (`CollisionEnter`). **`ANCHOR_RAISE` lowers the platform via a FORCE-CARVE** (not a buried additive paste): `anchorY = TOP_PLANE_Y + ANCHOR_RAISE` drops the circle that many blocks, and `carve_hollow` adds Empty cells over the footprint (the `ANCHOR_RAISE+1` rows above the ring); `pasteExit` then pastes with **`force=true`**, so the Empty cells DIG a clean recess instead of the circle burying under the terrain (the symptom of the old `force=false` + raised anchor: the prefab "did not spawn" because it was underground). `ANCHOR_RAISE=2` = a 2-block-deep sunken circle; 0 = flush. `pasteExit` also FORCE-LOADS the chunks around `ESCAPE` first (the gate-open reveal happens while the party is at the last shrine, so that chunk may be unloaded - a bare paste would no-op). 11x11 standable footprint; `ESCAPE_RADIUS` 6.0 covers it. Re-run the tool to regenerate.
- **The runtime beats (exit + cave shafts) RE-PASTE at +4s and +9s into PREP** (`OBJECTIVE_REPASTE_DELAYS_SEC`) to beat a worldgen-overwrite race: on a busy relaunch the first paste can land before its chunks finish generating and get clobbered. The cave re-carve is idempotent via `ChaseState.caveCarveY` (the stored surface Y -> the same carve, never a stacked shaft). **Surface shrines are worldgen-placed, so they are NOT in the re-paste loop** (worldgen does not race itself).

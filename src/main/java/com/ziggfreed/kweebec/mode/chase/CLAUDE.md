# mode/chase/ - the Chase round loop (Relight & Escape)

Router for `mode/chase/`. Driven 1 Hz from [`../../round/RoundStateMachine`](../../round/RoundStateMachine.java) on the instance world thread; stateless (all state lives on [`../../round/RoundInstance`](../../round/RoundInstance.java) + [`ChaseState`](ChaseState.java)).

- **[`ChaseMode.tick`](ChaseMode.java) is the whole loop**: PREP (a breath) -> RITUAL (spawn the hunter + heartbeat, survivors cleanse the shrine FURNACES by offering Moonbloom, corruption climbs with time + per shrine) -> ESCAPE (the last shrine opens the gate, the hunter hard-locks onto the nearest survivor, crossing the gate or reaching the exit wins). `lazyPlayerSetup` handles async arrival/disconnect via player ABSENCE (two grace windows: longer for still-teleporting, short for entered-then-dropped) + first-arrival HUD install + the dread music bed one-shot.
- **The shrine cleanse is an interactable FURNACE, WORLDGEN-PLACED + INTERACTION-DISCOVERED** (0.4.x rework): each shrine is a `KweebecNightmare_Shrine` furnace block (a 2x2 `Bench_Furnace` reskin) BAKED into a prefab - 3 surface (copies of vanilla `Cursed_Crypt`/`Incidental_Slate_Ruin` prefabs, placed at fixed grove spots via biome `List` props) + 2 runtime-carved cave shafts = 5 normal-mode. NO ring, NO runtime `setBlock` (a single setBlock of the 2x2 furnace model rendered as "a different block" - the bug; bake it in the prefab with the footprint clear). A survivor presses F to offer Moonbloom; [`../../interaction/ShrineSubmitInteraction`](../../interaction/CLAUDE.md) -> `ChaseState.shrineForBlock` LAZILY registers that shrine from the targeted block (no position pre-tracking), deposits up to `RuleSet.cleanseCost()` via `InventoryUtil.take`, and `ChaseMode.lightShrine` lights the furnace (green fire) when full, bumping corruption once. `handleShrines` is a per-tick lit RECONCILER over the DISCOVERED shrines (re-asserts the `"lit"` state via `world.setBlockInteractionState`; a backstop for a missed call / chunk reload). Cleansing is gated to RITUAL (no offering in PREP). The old channel risk (`loudestChanneller`/`feedbackStage`/`SHRINE_FLARE_AT` flare ladder) is vestigial; the hunter falls back to nearest-survivor pursuit. The reusable objective-as-block recipe (for the MMO too) is in the `hytale-modding` `worldgen-v2.md` skill.
- **Escape wins on EITHER trigger** (`checkEscapes`): reaching the exit pad (`ESCAPE_RADIUS_SQ`) OR simply CROSSING the gate (`pos.z <= GATE.z - 2`). The gate-cross is the robust trigger - once through the open archway a survivor has escaped even if the distant exit pad is hard to reach in the dark. The whole win chain (shrine lit, all-shrines/gate-open, escaped, nearest-to-exit) logs at INFO under a `[Kweebec][win]` tag. The gate is an open glowing archway (light blocks framing an opening, no blocking wall), not a door.
- **Survivor game mode is normalized to Adventure ONCE on entry, never per-tick.** `lazyPlayerSetup` sets Adventure only while `!st.isGameModeApplied()` (the hunter's marked target only holds on an Adventure survivor). Re-asserting it every tick fought an admin switching to Creative (it snapped back within ~1s - the 0.2.0 bug); [`../../hunter/AiHunterController`](../../hunter/AiHunterController.java) no longer touches game mode either. Do NOT re-add a per-tick game-mode force - an admin in Creative simply self-excludes from the hunter lock.
- **[`ChaseState`](ChaseState.java) is COUNT-BASED (not a pre-built shrine list) + holds the 0..1 corruption meter.** `totalShrines` = `ArenaLayout.SURFACE_WORLDGEN_SHRINES` (3) + `RuleSet.caveShrineCount()` (2) = the win denominator, known up front from the deterministic placement. A lazy `discovered` `Map<Vector3i,ShrineState>` fills as `shrineForBlock` registers each furnace on first F-press; `litShrines()`/`allShrinesLit()` count over it; `shrines()` returns the discovered values (for the reconciler). `caveCarveY` is per-cave carve-idempotency bookkeeping. `corruptionTier()` (0/1/2) drives heartbeat cadence + darkness; `hunterSpeed(rules)` maps corruption to a speed multiplier. [`ShrineState`](ShrineState.java) is one shrine's `blockPos`/`submitted`/`lit`/`litRendered` (its anchor is now just the synthesized block position). Shrine cleanse is interaction-driven (not proximity); only RESCUE uses `ArenaLayout.INTERACT_RADIUS_SQ`, escape uses `ESCAPE_RADIUS_SQ`. The `progress`/`channeller`/`caveSurfaceTopY` fields are vestigial.
- **Outcome resolution** (`resolveOutcome`): any escaped = `ESCAPED`; cap reached = `TIMED_OUT`; everyone present cocooned = `CAUGHT`; the party fully empties = `ABORTED`; else `null` (keep running). Death/cocoon itself is handled in [`../../death/`](../../death/CLAUDE.md), not here.

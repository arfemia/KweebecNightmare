# mode/chase/ - the Chase round loop (Relight & Escape)

Router for `mode/chase/`. Driven 1 Hz from [`../../round/RoundStateMachine`](../../round/RoundStateMachine.java) on the instance world thread; stateless (all state lives on [`../../round/RoundInstance`](../../round/RoundInstance.java) + [`ChaseState`](ChaseState.java)).

- **[`ChaseMode.tick`](ChaseMode.java) is the whole loop**: PREP (a breath) -> RITUAL (spawn the hunter + heartbeat, survivors cleanse the shrine FURNACES by offering Moonbloom, corruption climbs with time + per shrine) -> ESCAPE (the last shrine opens the gate, the hunter hard-locks onto the nearest survivor, and the party CO-OP EXTRACTS by holding the Heartwood platform together). `lazyPlayerSetup` handles async arrival/disconnect via player ABSENCE (two grace windows: longer for still-teleporting, short for entered-then-dropped) + first-arrival HUD install + the dread music bed one-shot.
- **The shrine cleanse is an interactable FURNACE, WORLDGEN-PLACED + INTERACTION-DISCOVERED** (0.4.x rework): each shrine is a `KweebecNightmare_Shrine` furnace block (a 2x2 `Bench_Furnace` reskin) BAKED into a prefab - 3 surface (furnace-baked `Shrine_Crypt_01/02`/`Shrine_Crypt_Large`/`Shrine_House_01/02` hosts SCATTERED by biome `Mesh2D` props, their count + spacing reconciled at round start by [`../../arena/ShrinePlacement`](../../arena/CLAUDE.md) - detect the worldgen markers, top up only the MISSING ones spaced) + 2 runtime-carved cave shafts = 5 normal-mode. NO ring, NO runtime `setBlock` (a single setBlock of the 2x2 furnace model rendered as "a different block" - the bug; bake it in the prefab with the footprint clear). A survivor presses F to offer Moonbloom; [`../../interaction/ShrineSubmitInteraction`](../../interaction/CLAUDE.md) -> `ChaseState.shrineForBlock` LAZILY registers that shrine from the targeted block (no position pre-tracking), deposits up to `RuleSet.cleanseCost()` via `InventoryUtil.take`, and `ChaseMode.lightShrine` lights the furnace (green fire) when full, bumping corruption once. `handleShrines` is a per-tick lit RECONCILER over the DISCOVERED shrines (re-asserts the `"lit"` state via `world.setBlockInteractionState`; a backstop for a missed call / chunk reload). Cleansing is gated to RITUAL (no offering in PREP). The old channel risk (`loudestChanneller`/`feedbackStage`/`SHRINE_FLARE_AT` flare ladder) is vestigial; the hunter falls back to nearest-survivor pursuit. The reusable objective-as-block recipe (for the MMO too) is in the `hytale-modding` `worldgen-v2.md` skill.
- **Escape is a CO-OP EXTRACTION HOLD** (`checkExtraction`, replaces the old single-survivor gate-cross win): once the gate is open, the whole REQUIRED survivor group must stand on the Heartwood platform (within `ArenaLayout.ESCAPE_RADIUS_SQ` of `ESCAPE`) TOGETHER for `RuleSet.extractionHoldSeconds()`; the hold RESETS the instant the group breaks (someone steps off or is caught). The required set is per-difficulty via `RuleSet.extractionMode()`: `ALL_MOBILE` counts only ACTIVE survivors (a cocooned teammate does not block; Amateur/Hardcore), `EVERYONE` also requires every still-RESCUABLE cocoon (so it blocks until revived) while EXCUSING a permanently-cocooned one (no soft-lock; Nightmare). The continuous-hold timer is the reusable **ziggfreed-common `instance/zone/ZoneHoldTimer`** primitive (pure logic, unit-tested) - `ChaseState` owns one, built in `onStart` from the rule-set; kweebec keeps only the required-set policy + the escape action. On completion every active survivor is marked escaped AT ONCE (never per-tick on contact - that would flip `isActive()` off and let the group cheese the hold un-hunted), so `resolveOutcome` sees `anyEscaped()` that tick. The win chain logs at INFO under `[Kweebec][win]`. The gate is an open glowing archway (light blocks framing an opening, no blocking wall), not a door; the exit pad is the `KweebecNightmare/Extraction_Pad` prefab (a vanilla Frost Elemental Circle, see [`../../arena/CLAUDE.md`](../../arena/CLAUDE.md)). The "X/Y on the platform" + hold countdown shows on the `#Extraction` HUD line (see [`../../feedback/CLAUDE.md`](../../feedback/CLAUDE.md)).
- **Shrine DISCOVERY markers (1.4.x):** the first F-press on a shrine registers a world-map / compass marker through the round's `MapDiscovery` (ziggfreed-common) in [`../../interaction/ShrineSubmitInteraction`](../../interaction/CLAUDE.md), BEFORE the Moonbloom gate (a found-but-unaffordable shrine is still marked); `ChaseMode.lightShrine` swaps it to a cleansed "done" icon (`Shrine.MARKER_ICON` -> `LIT_MARKER_ICON`). Per-difficulty via `RuleSet.shrineDiscovery` (trigger `OFF`/`ON_INTERACT`/`PROXIMITY`) x `shrineDiscoveryVisibility` (`PER_PLAYER`/`SHARED`): Amateur shared, Nightmare self, Hardcore off. The tracker is created + `attach`ed in `RoundService.onInstanceReady` (so the compass is on from round start). PROXIMITY is a ready seam (radius `shrineDiscoveryRadius`): register shrines up front + call `MapDiscovery.revealWithin` from `tick`; only ON_INTERACT is wired now.
- **Survivor game mode is normalized to Adventure ONCE on entry, never per-tick.** `lazyPlayerSetup` sets Adventure only while `!st.isGameModeApplied()` (the hunter's marked target only holds on an Adventure survivor). Re-asserting it every tick fought an admin switching to Creative (it snapped back within ~1s - the 0.2.0 bug); [`../../hunter/AiHunterController`](../../hunter/AiHunterController.java) no longer touches game mode either. Do NOT re-add a per-tick game-mode force - an admin in Creative simply self-excludes from the hunter lock.
- **[`ChaseState`](ChaseState.java) is COUNT-BASED (not a pre-built shrine list) + holds the 0..1 corruption meter.** `totalShrines` = `ArenaLayout.SURFACE_WORLDGEN_SHRINES` (3) + `RuleSet.caveShrineCount()` (2) = the win denominator, known up front from the deterministic placement. A lazy `discovered` `Map<Vector3i,ShrineState>` fills as `shrineForBlock` registers each furnace on first F-press; `litShrines()`/`allShrinesLit()` count over it; `shrines()` returns the discovered values (for the reconciler). `caveCarveY` is per-cave carve-idempotency bookkeeping; `surfaceShrinePositions` holds the resolved surface shrine world positions (worldgen-detected + topped-up) that `ShrinePlacement` publishes, so `ArenaBuilder.plantMoonbloom` clusters supply at the REAL shrines. `corruptionTier()` (0/1/2) drives heartbeat cadence + darkness; `hunterSpeed(rules)` maps corruption to a speed multiplier. [`ShrineState`](ShrineState.java) is one shrine's `blockPos`/`submitted`/`lit`/`litRendered` (its anchor is now just the synthesized block position). Shrine cleanse is interaction-driven (not proximity); only RESCUE uses `ArenaLayout.INTERACT_RADIUS_SQ`, escape uses `ESCAPE_RADIUS_SQ`. The `progress`/`channeller`/`caveSurfaceTopY` fields are vestigial.
- **Outcome resolution** (`resolveOutcome`): any escaped = `ESCAPED`; cap reached = `TIMED_OUT`; everyone present cocooned = `CAUGHT`; the party fully empties = `ABORTED`; else `null` (keep running). Death/cocoon itself is handled in [`../../death/`](../../death/CLAUDE.md), not here.

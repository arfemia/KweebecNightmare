# Changelog

Developer changelog for Kweebec Nightmare. User-facing release notes live in `patch-notes/`.

## 0.6.0 (in development)

PvP pass: two new player-versus-player modes (Kweebec Clash team brawl + Kweebec Domination control point) running on the same instance/round engine as the co-op chase, a Kweebec Sapling reskin for combatants, throwable knockback/stun mushroom pickups, a new host NPC + dialogue, and asset-driven PvP presets/arenas. Built green (java + ziggfreed-common); the mode loops, the reskin, the pickups, the NPC, and the presets/arenas validate in-game.

- **Two PvP modes over the shared round engine.** `KweebecMode`/`ModeRegistry` gain a Clash (team brawl) and a Domination (control point) mode beside the co-op Chase; each seats `teamSize * 2` players into a sealed arena instance and resolves through the generic, engine-free `ziggfreed-common` match resolvers (`instance/match/WinConditionResolver` for Clash, `instance/zone` control state for Domination), so the win precedence (team-wipe gate, score cap, timer compare, sudden-death overtime, draw) lives in the shared lib, not the mode.
- **PvP stakes as flat `RuleSet` knobs (grouped views `clash()` / `domination()`).** New shared PvP fields (`teamSize`, `friendlyFire`, `modelSwapId`/`modelSwapScale`, `arenaId`/`arenaTag`) plus the Clash knobs (`WinCondition`: `LAST_TEAM_STANDING`/`MOST_HITS_LANDED`/`MOST_KILLS`/`TDM_SCORE_TO_KILLS`; `RespawnPolicy`: `NONE`/`LIMITED`/`INFINITE`; `respawnDelaySeconds`/`maxLives`/`scoreToWin`/`suddenDeathSeconds`, the mushroom-wave cadence/count/cap/cycle) and the Domination knobs (`scoreToWin`/`pointHoldSeconds`/`accrualPerSecond`/`pointRadius`/`contestRule`/`captureNeutralizes`/`respawnDelaySeconds`), all folded by the same `defaults < pack < owner < runtime` model and exposed as immutable grouped views (`ClashConfig`/`DominationConfig`).
- **Kweebec Sapling reskin for combatants.** A PvP round swaps each player's model to `Kweebec_Sapling` (`RuleSet.modelSwapId`/`modelSwapScale`), so both teams fight as the grove's own saplings.
- **Throwable knockback / stun mushroom pickups.** PvP arenas seed harvestable grove-throwable pickups (the existing Gustbloom knockback + a stun variant) on a configurable wave cadence (`mushroomCadenceSeconds`/`mushroomWaveCount`/`mushroomMaxAlive`/`mushroomCycle`), reusing the proven `GroveThrowable` gather/throw/burst chain.
- **New PvP host NPC + dialogue.** A grove warden NPC with a branching-dialogue entry lets a player choose Clash or Domination, the team size, and the difficulty before launch (the diegetic trigger into the existing Enter-the-Nightmare flow).
- **Asset-driven PvP presets + arenas.** PvP presets are authored on `RoundPresetAsset` (the same Pattern-A codec as the chase presets) and the arenas come from the `ziggfreed-common` arena catalog (pinned by `arenaId` or chosen by `arenaTag`, defaulting to the mode id), so the offering is pack-retunable with zero new Java.
- **`ClashPresetValidator` config-surface guard.** A pure, engine-free `round/ClashPresetValidator.validate(RuleSet)` the lobby owner runs before a PvP round to surface contradictory presets (unsupported `teamSize`, `INFINITE` respawn with a `LAST_TEAM_STANDING` elimination win, a score-capped condition with `scoreToWin <= 0`, a non-elimination round with no timer, a negative sudden-death window), each a one-line `[Kweebec]` warning.
- **Fixed: a wedged round no longer traps a downed player under the infinite cocoon.** A 1 Hz tick that keeps THROWING (the trigger seen in the field was a runtime linkage error, `NoClassDefFoundError` on the bundled-api `RoundCompletedEvent$Outcome` enum from a stale incremental jar, hit only on the caught-player branch) silently swallowed its outcome, so the round never resolved; the cocoon is an `Infinite` `DisableAll` effect cleared ONLY on resolve/eject, so a solo player who died was frozen dead with the round never ending. The state machine gained a watchdog: after `MAX_CONSECUTIVE_TICK_FAILURES` (5) consecutive throwing ticks it force-evacuates the round via the new `RoundService.forceAbort`, which revives + ejects every player (clearing the cocoon on overworld return) and tears the instance down WITHOUT touching `RoundCompletedEvent.Outcome` (so it recovers even when that enum is the unloadable class). The tick-failure log gained the exception type so a linkage error is no longer hidden behind a bare `getMessage()`. A clean rebuild also re-bundles a loadable `Outcome` so the normal path resolves `CAUGHT` again.

Technical: built green via `gradlew build` (java + ziggfreed-common). The two mode loops, the Sapling model swap, the mushroom-pickup waves, the host NPC role/dialogue, the win/capture resolution feel, and the asset-driven presets/arenas are NOT gradle-validated - they need an in-game pass (a 1v1 and a 2v2 Clash to each win condition + respawn policy, a Domination capture/hold/neutralize cycle, friendly-fire on/off, an arena pin vs tag pick).

Score-tiered chase win rewards: the end-of-game spoils are now an asset-driven, per-difficulty loot table that pays better the better you played, replacing the flat per-preset Moonbloom handout. Built green (java + ziggfreed-common); the roll + claim surfaces validate in-game.

- **Per-difficulty loot tables, rolled by run score (replaces the flat `Rewards` list).** Each chase instance preset now references a `ziggfreed-common` loot table by `RewardTableId` (`Server/ZiggfreedCommon/LootTables/Chase_{Amateur,Nightmare,Hardcore}.json`): a guaranteed staple plus a weighted pool whose pick count and premium unlocks scale with the player's `PlayerScore.total()`. Amateur is modest (1 base roll, cap 3), Nightmare richer (2, cap 5), Hardcore the best (3, cap 6), with the rarest entries (concentrated life essence) gated behind a high score. WIN-ONLY (the existing `RewardOnExit.ON_WIN`); a lost run still grants nothing.
- **Sensible loot: life essence + the full glow-throwable arsenal.** Pools hand out vanilla `Ingredient_Life_Essence` (and `Ingredient_Life_Essence_Concentrated` at high score) alongside an assortment of the mod's own throwables (Moonbloom / Gustbloom / Mirebloom / Emberbloom) in per-difficulty quantity ranges, so a good run restocks your kit. Reward chips auto-label from each item's own display name, so no new localization.
- **Rolled once at resolve, where the score is known; claimed as before.** `KweebecExperience.stashResults` rolls the table per player with a deterministic per-(round, player) seed, persists the CONCRETE rolled list to the durable `PendingRewardStore` (no grant), and stashes the same list for the results-page chip preview, so the preview matches the eventual payout exactly. Delivery is unchanged: claimed back in the overworld through the block-first full-inventory guard. The in-arena crypt chest stays on Hytale's native loot format (best practice for in-world loot), untouched.

Technical: built green via `gradlew build` (kweebec) + `ziggfreed-common` (0.3.2, the new `LootTable`/`LootEntry`/`LootTableAsset` primitive + `InstancePreset.rewardTableId`). The score thresholds + pool weights are tunable data; the roll variety, the per-difficulty richness, the score->loot scaling, and the chip-matches-payout preview are NOT gradle-validated - they need an in-game pass (a low-effort win vs a fast full-cleanse win on each difficulty, a claim with a full inventory).

Per-difficulty thrown-mushroom damage: a generic, asset-driven damage table so each authored glow-throwable hits for a per-preset amount with zero per-item Java. Built green (java).

- **New `ThrowableDamage` preset knob (a `Map<DamageCauseId, amount>`).** `RoundPresetAsset` gains a `ThrowableDamage` object field (decoded via `MapCodec<>(Codec.DOUBLE, ...)`), folded into a new immutable `RuleSet.throwableDamage()` map. When a thrown glow-throwable bursts on a mob in a round, `KweebecDamageSystem` reverse-resolves the hit's custom `DamageCause` id (`DamageCause.getAssetMap().getAsset(index).getId()`) and, if the active preset tunes it, overrides the burst's authored `BaseDamage` with the per-difficulty value. ONE mechanism covers Moonbloom, Emberbloom, and any future authored throwable with no new Java; a cause absent from the table (a melee hit, or a throwable the preset does not tune) is left untouched, and outside a round the burst's authored damage stands.
- **Authored values.** Moonbloom (`KweebecNightmare_Moonbloom`): Amateur 25 / Nightmare 12 / Hardcore 1 (it still stuns on top of the damage). Emberbloom (`KweebecNightmare_EmberHit`): item base lowered to 25 in the burst, overridden to Amateur 150 / Nightmare 125 / Hardcore 100 against the boss + mobs (the survivor friendly-fire guard still nulls it on a teammate). The override applies to the co-op chase mobs/boss only; PvP keeps Moonbloom as a zero-damage stun utility.

Technical: built green via `gradlew build` (java). The per-difficulty damage feel is NOT gradle-validated - it needs an in-game pass (throw a Moonbloom + an Emberbloom at a hunter/boss on each difficulty; confirm a teammate splash is still harmless).

## 0.5.0 (in development)

Co-op extraction rework: the Chase no longer ends the moment one survivor slips through the gate. The whole required survivor group now has to HOLD the Heartwood platform together to escape, the exit is a real vanilla ritual circle instead of a plain light pad, and a new HUD line tracks the hold. Built green (java + ziggfreed-common, 82 common tests); the hold logic + prefab + HUD validate in-game.

- **Group extraction hold (replaces the single-survivor gate-cross win).** `ChaseMode.checkExtraction` supersedes the old `checkEscapes`: once the gate opens, the required survivor group must stand TOGETHER on the Heartwood platform (within `ArenaLayout.ESCAPE_RADIUS_SQ`) for `RuleSet.extractionHoldSeconds()`, and the hold resets the instant the group breaks (someone steps off or is caught). Every active survivor is marked escaped AT ONCE on completion (never per-tick on contact, which would flip `isActive()` off and let the group park on the pad un-hunted), so `resolveOutcome` still resolves `ESCAPED` via `anyEscaped()`. The old `pos.z <= GATE.z` crossing trigger is gone.
- **New shared primitive (lifted to `ziggfreed-common` 0.3.2): `instance/zone/ZoneHoldTimer`.** The continuous "required group holds a zone TOGETHER for X seconds, reset on break" state machine is a PURE, engine-free primitive (caller injects `nowMillis` + the live counts, like `EncounterDirector`), unit-tested, reusable for any extraction pad / capture point / king-of-the-hill (the MMO dungeons/raids are the target). `ChaseState` owns one, built in `onStart`; kweebec keeps only the required-set policy + the escape action. Per the paradigm: the reusable timer is lifted, the mod-specific policy stays in the consumer.
- **Per-difficulty extraction knobs (`ExtractionMode` + `ExtractionHoldSeconds`).** New `RuleSet`/`RoundPresetAsset` knobs (`ExtractionMode`: `ALL_MOBILE` counts only mobile survivors so a downed teammate does not block; `EVERYONE` also requires every still-rescuable cocoon, blocking until revived, while excusing a permanently-cocooned one so it can never soft-lock). Amateur ALL_MOBILE / 4s, Nightmare EVERYONE / 6s, Hardcore ALL_MOBILE / 8s, authored in `DefaultPresets` + the preset JSONs.
- **The exit is a vanilla Frost Elemental Circle with a no-op void portal.** `ArenaBuilder` pastes `KweebecNightmare/Extraction_Pad` (the platform) at the escape: a vanilla `Monuments/Unique/Elemental_Circles/Frost/Middle` medallion copied into the pack by the new `tools/build_extraction_pad.py` (vanilla prefab keys do not resolve at runtime, so it is committed like `Corrupted_Well`), stripped to its solid blocks (no air / encounter-spawner / construction-signs / water). The tool KEEPS the full elemental circle (incl. its central crystal medallion) and crowns the centre with the mod's own glowing `KweebecNightmare_Moonbloom_Plant` as the exit marker (no hand-built portal blocks - the plant is placed only on full-face column tops so it has the `Support.Down=Full` it needs to render). The platform is pasted FLUSH (`anchorY=13`) because the additive `force=false` paste buries a sunk platform (ring under terrain) and walls-up a raised one, and it is revealed at the gate-open beat via `ArenaBuilder.pasteExit` (NOT at round start - `pasteObjectives` now only carves the cave shafts). The old 3x3 orange-light `Exit.prefab.json` AND the old purple/blue `Gate.prefab.json` light archway are removed (`ArenaBuilder.pasteGate`/`GATE_PREFAB` gone); `ChaseMode.openGate` now fires the gate-open beat logically without pasting an arch.
- **New extraction HUD line + cues.** A `#Extraction` HUD element shows the live "X/Y on the platform" count plus the hold countdown during the gate-open climax (blank otherwise); a toast cues the party when the hold first begins; the gate-open banner and the escape objective line now read "gather on the platform together".

Technical: built green via `gradlew build` (kweebec) + `ziggfreed-common` (0.3.2, the `ZoneHoldTimer` primitive, 82 tests). The `.ui` element, the prefab paste/anchor flush-snap, the per-difficulty hold feel, and the cocoon-blocking EVERYONE mode are NOT gradle-validated; they need an in-game multi-player pass (a solo hold, a 2-4p hold with a teammate stepping off, a Nightmare run with a cocooned-then-rescued teammate, a permanently-downed Hardcore teammate not soft-locking).

Glow-throwable arsenal + a redone crypt chest: three new thrown glowing-mushroom variants (knockback / slow / damage) modeled on Moonbloom, a data-driven grove-throwable asset type, boss-phase damage clusters, and the dungeon-crypt chest pool rebuilt around them. Built green (java + ziggfreed-common); the throw / effect / loot surfaces validate in-game.

- **Three glow-throwable variants, effects 100% asset-authored (no Java).** `Gustbloom` (vanilla Glowing Blue mushroom -> knockback via `DamageEffects.Knockback` `Type:Point`), `Mirebloom` (Glowing Green -> slow via an `ApplyEffect` step + the `KweebecNightmare_MireSlow` EntityEffect), `Emberbloom` (Glowing Orange -> real AoE `BaseDamage`). Each reuses the proven Moonbloom chain verbatim (item -> `Throw` -> `Projectile_Config_*` over `Projectile_Config_Bomb_Base` -> AOECircle `Burst` -> custom `Physical` DamageCause) and the full gather loop (`_Plant` block -> `_Drop` droplist -> one-block cluster prefab). The engine already exposes all three effects, so the ONLY Java is the shared id authority `moonbloom/GlowThrowables` + the Emberbloom friendly-fire guard. Configurable per variant via the burst fields (`Force`/`VelocityY`, `HorizontalSpeedMultiplier`/`Duration`, `BaseDamage`, AOECircle `Range`).
- **Data-driven grove distribution (a new pack-authorable asset type).** `asset/GroveThrowableAsset` + `GroveThrowableConfig` (`Server/KweebecNightmare/GroveThrowables/*.json`: `Enabled`/`PrefabKey`/`PerShrineCount`/`ScatterCount`/`RespawnWithWaves`/`MinCorruptionTier`) generalize the hardcoded Moonbloom scatter. `ArenaBuilder.plantGroveThrowables` iterates `placeable()` (the generalized `plantClusters` over each entry's prefab), driven for the initial supply from `ShrinePlacement` + each Moonbloom respawn wave from `ChaseMode`. ZERO jar defaults (pure pack content); the Gust/Mire entries ship `Enabled:false`, so the full gather loop is present but DORMANT until a pack flips it. Add a variant = a new JSON, zero Java. ADD-only store (no Control key), like Mutators; registered in `KweebecAssetRegistrar`.
- **Boss-phase Emberbloom clusters (the reusable seam in ziggfreed-common).** `BossController` rings harvestable Emberbloom clusters around the Warden on each phase and REPLENISHES them on a configurable timer, so survivors gather + throw them at the boss to keep a 600/450/300-HP capstone killable. All data on the shared `common.instance.encounter.MultiPhaseBossAsset` (new `ThrowableClusterId` / `Phase{1,2,3}ThrowableCount` / `ThrowableRingRadius` / `ThrowableRespawnSeconds` knobs; requires ziggfreed-common 0.3.2), placed via `SpawnPlacement`-style ring math + the generalized `ArenaBuilder.plantClusterRing`. Authored on `Warden.json` (4/5/6 per phase, 8-block ring, 20s respawn). Any future boss reuses it.
- **Friendly-fire guard (Emberbloom).** Vanilla AoE has no asset-level attitude filter, so `KweebecDamageSystem` nulls Emberbloom-cause damage on a survivor victim (mirrors the existing Moonbloom-splash skip); the boss + mobs take it normally. Gust/Mire ship disabled, so their friendly-effect filter is an enablement TODO.
- **Dungeon-crypt chest pool rebuilt around the throwables.** `KweebecNightmare_Crypt_Loot` (the droplist behind the `Furniture_Dungeon_Chest_Epic` chest in `Shrine_Crypt_Large`) is redone as a glow-throwable cache: guaranteed Gustbloom + Mirebloom + a smaller Emberbloom roll, plus a weighted bonus roll. The chest is INDEPENDENT of the grove-distribution flag, so descending the large crypt is the reliable way to stock all three even while the grove scatter stays disabled. Replaces the old Moonbloom + potion/ore contents.

Technical: built green via `gradlew build` (kweebec) + `ziggfreed-common` (0.3.2, the boss-asset extension). The item / burst / effect / plant / droplist JSON, the knockback feel (Point knockback is shooter-sourced, so Gustbloom shoves away from the thrower not radially), the boss killability tuning, and the crypt loot are NOT gradle-validated - they need an in-game pass (`/give KweebecNightmare_{Gustbloom,Mirebloom,Emberbloom} --quantity=N`, a Hardcore boss run, a crypt-chest open). Design + the verified asset capabilities: the parent-repo plan note `need-to-author-additional-wobbly-hickey`.

Hunter fear pass: a scarier hunter look, vocal taunts, and far harder per-archetype attacks. Built green (java); the asset + audio surfaces validate in-game.

- **Glowing-eyed corrupted appearance.** A new pack model `KweebecNightmare_HunterCorrupted` (`Server/Models/KweebecNightmare/`) extends the vanilla `Kweebec_Sapling_Razorleaf` (keeps the spear silhouette), swaps to the darkest vanilla Kweebec body texture as a corruption stopgap, and attaches a persistent `Spectre_Void_Hands` glow particle to each eye bone (`Particles[]` with `TargetNodeName`, the verified vanilla `Golem_Guardian_Void` eye-glow mechanism). All four hunter Variant roles (`KweebecNightmare_{Blight,Lunger,Spitter,Ambusher}`) point their `Appearance` at it. The eye-bone node name (`L-Eye-Attachment`/`R-Eye-Attachment`) and a bespoke dark-body + white-eye PNG swap are the in-game verification / pending-art items (a wrong bone name no-ops the particle, no crash).
- **Snicker cues (two triggers, vanilla SFX, no custom audio).** (1) **On proximity** - `ScareDirector.tick` rolls a throttled (~5s/player), ~12%-per-tick chance once a survivor is in proximity band >= 2, playing an eerie cue at the NEAREST hunter's position, private to that survivor (mirrors the whisper layer but located ON the hunter, not mislocated). (2) **On hit** - `KweebecDamageSystem` rolls ~25% when a hunter lands a hit, played at the hunter's position, private to the victim. The attacker is identified as a hunter by reading its model asset id off the damage source ref via `ziggfreed-common` `EntityIdentifierUtil.getMobId` (the same self-contained technique the MMO uses for kill/XP attribution) and matching the `KweebecNightmare` prefix - no custom DamageCause needed. Sound id `SFX_Emit_Temple_Wisps` (one-constant retune point); resolution degrades to silence via `AssetIndexCache` if absent.
- **Per-archetype devastating melee.** Each archetype template overrides the spear attack's `Melee_Damage` var (top-level `InteractionVars.Melee_Damage`, the vanilla `Bison` lever) with a flat Absolute Physical + a stronger launch knockback, up from the vanilla NPC melee (5 dmg / force 1): Lunger 14/8, Stalker (Blight) 10/4, Ambusher 9/4, Spitter 7/3 (`Physical`/`Knockback.Force`). Impact sound/particle fx inherited from `NPC_Attack_Melee_Damage`. Hunter melee stays plain physical, so damage-taken scoring is unaffected.

Technical: built green via `gradlew build` (java). The new model JSON, the role/template asset edits, the eye-bone particle, and the snicker SFX are NOT gradle-validated - they need an in-game `/kweebec start` pass (eye-bone name, emissive read, snicker audibility/throttle, per-archetype damage feel, no one-shot of a fresh survivor).

## 0.4.0 (in development)

The Moonbloom loop: a gathered glowing-mushroom RESOURCE with two competing sinks (cleanse vs defend), plus per-player scoring, a native scored event, and a persisted leaderboard. Built green (java + api); the throw/gather/cleanse surfaces validate in-game.

- **New shared primitive (lifted to `ziggfreed-common` 0.3.0).** `inventory/InventoryUtil` - `give`/`count`/`take`/`spend`/`has` a resource item by id across the combined inventory sections (`InventoryComponent.getCombined(..., EVERYTHING)`), world-thread, try-guarded. A generic Hytale primitive per the paradigm; Kweebec consumes it (cleanse spend, the `give` command), never re-derives it. Requires the rebuilt `ziggfreed-common` jar (gradle pin bumped to 0.3.0).
- **Moonbloom resource (asset-driven gather).** A custom charge item `KweebecNightmare_Moonbloom` (reskin of the vanilla `Weapon_Bomb_Stun` so it inherits a throwable Primary) drops from a harvestable grove block `KweebecNightmare_Moonbloom_Plant` via `Gathering.Harvest.DropList` -> `Server/Drops/KweebecNightmare/KweebecNightmare_Moonbloom_Drop.json`. `ArenaBuilder.plantMoonbloom` stamps a guaranteed cluster at each unlit surface shrine + a seed-deterministic grove scatter (a 1-block `KweebecNightmare/Moonbloom` prefab pasted per-column floor-snapped, reusing the proven paste path), and `ChaseMode.maybeRespawnMoonbloom` regrows a configurable batch at the configured mid-match wave times.
- **Cleanse = an interactable shrine FURNACE.** Each grove shrine is a custom furnace-reskin block (`KweebecNightmare_Shrine` - the vanilla `Bench_Furnace` model with a green-fire `State.Definitions.lit`) placed at each surface anchor (REPLACING the old light-pillar prefab) and in each cave chamber. A survivor presses F to OFFER Moonbloom: a registered `SimpleInstantInteraction` (`ShrineSubmitInteraction`, the mod's first custom block interaction) resolves the shrine from the F-target block and deposits up to its remaining need (`RuleSet.cleanseCost()` total, authored per preset in the shipped JSON - 3 on Amateur) via `InventoryUtil.take`; once full, `ChaseMode.lightShrine` lights the furnace with green fire (`world.setBlockInteractionState(pos, bt, "lit")`), bumps corruption once, and announces it grove-wide. `ChaseMode.handleShrines` is now a per-tick lit RECONCILER (a cheap backstop that re-asserts the block state); the proximity-spend scan is gone. Cleansing is gated to the RITUAL phase (no offering during the PREP breather). The old `ShrineState` channel `progress`/`channeller` fields stay vestigial; `submitted`/`blockPos`/`litRendered` are added.
- **Throw-to-stun (asset chain + Perfect Utils + one damage hook, WORLD-AGNOSTIC).** Left-click throws a Moonbloom (a reskin of the vanilla `Weapon_Bomb_Stun`, so it inherits a throwable Primary): `Moonbloom_Throw` -> `Projectile_Config_KweebecNightmare_Moonbloom` -> `Moonbloom_Burst` handles flight + AOE. The item needs `MaxStack` to stack and a `{Type:ModifyInventory, AdjustHeldItemQuantity:-1}` step to consume on throw (vanilla bombs lack both). The burst deals a 1-HP tag carrying a **custom DamageCause** `KweebecNightmare_Moonbloom` (a pack-authored `Server/Entity/Damage/*.json`). A new unified `event/KweebecDamageSystem` (a `DamageEventSystem` in `getFilterDamageGroup()`) identifies a Moonbloom hit by that cause ALONE (`damage.getDamageCauseIndex()`) - source- and world-independent, because the burst damage is sourced from the SHOOTER, not the projectile (`DamageEntityInteraction` uses `context.getOwningEntity()`), so it cannot be told from a melee hit otherwise. On a Moonbloom hit to ANY non-player mob anywhere (overworld or round) it applies the Perfect Utils `StunMobAPI` freeze + the vanilla `Stun` visual (Perfect Utils' `DU_Entity_Stunned` has no particle); MOBS-ONLY (no teammate friendly-stun), credited to the thrower's `mobsStunned` when they are in a round. The SAME system accumulates real damage TO a survivor onto `PlayerRoundState.damageTaken` for scoring. `RuleSet.throwMode` selects PROJECTILE (default) or the scaffolded-but-unwired CONE fallback.
- **Configurable scoring.** `score/ScoringConfig` (baseline + par-time/speed + damage-budget/caution + per-stun bonus, immutable, runtime-overridable via `KweebecNightmareAPI.overrideScoring`/`scaleScoring`/`resolveScoring`) + a pure `score/ScoreCalculator.compute(...)` -> `api/PlayerScore`. `RoundService.resolve` computes each participant's score (a player's win = the round won AND they personally escaped).
- **Native scored event + leaderboard.** A new additive `api/KweebecRoundScoredEvent` carries the per-player `PlayerScore` map (fired AFTER `RoundCompletedEvent`, the MMO capstone reward hook). `score/Leaderboard` buckets best score + best winning time by PARTY SIZE, UUID-keyed, loaded at setup from `getDataDirectory()/leaderboard.json` and flushed off-thread + debounced via `FileUtil.writeStringAtomic` (Gson, `.bak` fallback). No UI; `forPartySize` is the query seam.
- **Config + commands.** `RoundPresetAsset` gains pack-authorable `CleanseCost`/`StunDurationMs`/`ThrowMode`/`MoonbloomPerShrine`/`MoonbloomScatter`/`MoonbloomRespawnCount`/`MoonbloomRespawnAtSeconds` (mapped onto the unchanged `RuleSet` builder). `/kweebec give [n]` (testing), `score`, and `leaderboard [partySize]` (the no-UI surface) added.
- **Paradigm:** generic inventory plumbing went to `ziggfreed-common`; the scoring value object went to the `api` module so the event carries it across the jar boundary; every new tunable is a `RuleSet` knob fed by the existing four-tier (`defaults < pack < owner < runtime`) model.

Technical: built green via `gradlew build` (java + api modules). The item / drop / block / projectile / interaction JSON and the throw->impact->stun + harvest->drop->pickup flows are NOT gradle-validated - they need the in-game spikes in the plan note `kweebec-nightmare-0-4-0-moonbloom-loop`.

## 0.3.0 (in development)

Cycle-3 feature pass building on the 0.2.0 foundation: deep worldgen prefab use, per-round procedural variety, and a horror toolkit. Three tracks landed (build green); all surfaces are pending live playtest (the worldgen biome + asset JSON validate only in-game).

- **Track A - worldgen prefab depth.** A build-time corruption-repaint generator (`tools/repaint_kweebec_prefab.py`) reads a native Kweebec structure prefab and swaps its palette to Void-blight blocks (deterministic per cell, validated against the block-id set), producing the committed `Corrupted_*` set (Well/Lamppost/Shop/GuardTower/House_Small/Bridge). The corrupted village ruins scatter through the outer grove via a sparse biome `Type:Prefab` prop (`LoadEntities:false`) and stand at gameplay anchors via a data-driven, surface-snapped `ArenaBuilder.pasteStructures` (roles: cover / chokepoint / beacon / landmark) gated by a shared `ExclusionMask`. The well centerpiece is repainted to `Corrupted_Well`. The placement table is asset-driven (`StructurePlacementAsset` + `StructureConfig` + `DefaultStructures`, one placement per `Server/KweebecNightmare/Structures/*.json`); `ExclusionMask` stays Java.
- **Track B - procedural variety.** Each round auto-seeds a fresh world (the hardcoded `instance.bson` `Seed` was removed; `WorldConfig.seed` defaults to `currentTimeMillis()` per fresh load), read back into `RoundInstance.worldSeed` and used to seed the shrine-ring rotation, the cave-anchor subset, and the structure selection from ONE coherent seed. A stackable `MutatorAsset` (Pattern A, additive deltas) folds into `PresetConfig.resolve` between the defaults<pack<owner fold and the runtime SCALE tier; jar mutators Swarm/Relentless/Caverns/Decay/Blitz ship, plus four new presets (Endless/Swarm/Pitch/Blitz). `RoundCompletedEvent` carries a `difficultyScore` (`DifficultyScore.compute`) for an installed MMO to scale rewards.
- **Track C - horror.** Three new hunter archetypes (Lunger charge, Spitter ranged-kite, Ambusher teleport-blink) authored as Variant roles of new sibling templates cloned from the de-leashed Blight template (the Attack-state chase instruction replaced per archetype, with the de-leashed Seek chase retained as the always-on fallback); `AiHunterController` is generalized from one role to a weighted per-hunter roster read from `HunterArchetypeConfig`, with per-hunter pace bands and corruption/`SpawnTier`-gated eligibility. A new `feedback/ScareDirector` (driven 1 Hz from `ChaseMode`) fans a per-survivor proximity dread vignette (`KweebecNightmare_Dread_1/2/3`, hysteresis), a throttled catch/near-miss jumpscare (camera shake + `SFX_Hedera_Scream` + a hard `KweebecNightmare_Jumpscare` tint + an EventTitle), a whisper/false-cue layer (`SFX_Emit_Forgotten_Whispers`, frequency rising with tier), and a corruption-tier music escalation in `MusicBedService`. The tier/band -> effect mapping is asset-driven (`ScareBeatAsset` + `ScareBeatConfig` + `DefaultScareBeats`). The pack-shipped `ScreenEffect` PNG question resolved positive (the vignettes use the verified-present vanilla `ScreenEffects/Fire.png`).
- **Paradigm:** every new tunable/content type is a pack-authorable `AssetBuilderCodec` (Pattern A), runtime-overridable - `MutatorAsset`, `StructurePlacementAsset`, `ScareBeatAsset` join the existing `RoundPresetAsset`/`HunterArchetypeAsset`.
- **Dialogue (shared-engine consumer).** Kweebec consumes the new `ziggfreed-common` branching-dialogue engine: a `DialoguePage` with branching nodes, an in-memory per-round flag store (`KweebecDialogueFlags`), the `kweebecnightmare.` i18n namespace, and a context-aware name header - all-default seams, zero MMO dependency. The engine, page, and `.ui` all live in the shared lib (consumed, never re-derived); the diegetic entry NPCs (the Grove Warden guide, the clash host) route the same page.
- **Worldgen BOTH-SEAMS pass + surface-snap foliage fix.** Runtime-pasted beats (the spiral / ladder cave shafts, surface shrines, exit, structures) were anchoring on the worldgen-decorated dead-tree canopy (trunks/leaves) instead of the ground: `ArenaBuilder`'s `SurfaceProbe` runs AFTER the worldgen prop/tree phase and stopped on the first opaque block. Fix: `ziggfreed-common` `SurfaceProbe` gains skip-set overloads + a new `BlockTypeLists` helper (resolves the engine's authored `BlockTypeList` assets `TreeWoodAndLeaves` + `AllScatter` to a key set); `ArenaBuilder` now probes past that decoration on both paste paths, landing every beat on the real surface. The cave-shrine anchors spread from a tight r~19-21 cluster to r~38 across the four quadrants (`ExclusionMask` auto-tracks). This establishes the both-seams split: gameplay objectives stay runtime-pasted (their worldgen positions are not recomputable - no placement guarantee, no runtime registry), decoration goes through worldgen. **Full atmospheric worldgen pass** on the grove biome: the `KN_Trees` palette widened to the full dead/ash/petrified/burnt-stage set plus rare Ash_twisted Large/Giant landmarks, plus two new scatter layers - `KN_Debris` (fallen logs / stumps / petrified logs / mossy boulders) and `KN_Graves` (Cursed_Crypt tombstones + sign-post, rare). All new prop layers are cloned from the proven `Corrupted_*` Prefab-prop shape with every prefab path verified to resolve; the big pit-carving `Crypt_Entrance` set-piece is kept off random scatter (reserved for the `ExclusionMask`-protected runtime seam). Requires the rebuilt `ziggfreed-common` jar. Research + the both-seams rationale: the parent-repo note `kweebec-worldgen-both-seams` (the new `ziggfreed-common` primitives are documented in its `world/CLAUDE.md` router).

Technical: built green via `gradlew build` (java + api modules). The biome JSON, NPC role JSON, and EntityEffect JSON are NOT gradle-compiled - they validate in-game; the in-game test gates are tracked in the plan note `kweebec-nightmare-cycle3-prefab-variety-horror`.

## 0.2.0 (in development)

Chase MVP ("Relight & Escape"). The full co-op round runs end to end against the installed Hytale server; the first live playtest surfaced three bugs (now fixed, see "First playtest fixes" below), and the remaining in-game behaviours are tracked as a handoff checklist.

- **Phase 1 foundation pass (post-MVP).** (1) **Shared primitives**: a new sibling mod `ziggfreed-common` (a HARD dependency now) holds the mod-agnostic engine wrappers (3D sound, camera, asset-index cache, `SurfaceProbe`, notifications, HUD helpers); `HeartbeatService` consumes its `Sound3D`. (2) **Asset-driven config**: the `RoundPreset` enum is replaced by pack-authorable `RoundPresetAsset` + `HunterArchetypeAsset` custom asset types (Pattern-A codecs) folded `defaults < pack < owner` by `PresetConfig`/`HunterArchetypeConfig`, with a 4th runtime tier via a new `KweebecNightmareAPI` (an installed MMO scales/overrides difficulty); `RuleSet` stays the runtime value object. (3) **Worldgen overhaul**: the flat play disc is removed for natural rolling terrain (the finite edge cliff stays), spawn kept deterministic by a small clearing, ambient trees moved to a biome `Type:Prefab` scatter (the runtime foliage ring is gone), and all runtime placement (shrines, exit, gate, cave shafts, hunter, a Kweebec-well centerpiece) floor-snaps to the real surface via `SurfaceProbe`; cave shafts re-point their chamber Y to the local surface. New `InventoryMode`/`RewardOnExit` knobs are defined (Java seam pending). Worldgen playtest fixes: a tree-prop `Jitter` 0.6 -> 0.5 (a >0.5 value failed BiomeAsset validation and aborted the load), PascalCase preset/hunter asset filenames, the tree prop switched to the proven vanilla tree-scatter pattern, and cave-shaft re-paste made idempotent (no more 3x stacking).

- **Round engine.** A 1 Hz `RoundStateMachine` over a `RoundService`/`RoundRegistry`/`RoundInstance` model owns party/role/lifecycle state (the engine has no party API). Each round is its own Path A engine instance: `InstancesPlugin.spawnInstance` from a pack-authored `instance.bson`, players entered via `teleportPlayerToLoadingInstance`, left via `exitInstance`, with a `CleanupTicker` backstop over the authored `RemovalConditions`.
- **Configurable RuleSet + presets.** Immutable `RuleSet` knobs (rescue style, lives, bleed-out, hunter count/speed, shrine count, round length, corruption ramp) with shipped presets Amateur / Nightmare / Hardcore. This is the replayability pillar; an installed MMO Skill Tree can select or scale them.
- **Chase mode.** `2 + partySize` grove shrines with hold-to-relight (channelling pins the survivor); a corruption meter rising with time + per shrine drives the heartbeat tier and darkness; the final shrine opens the Heartwood Gate and locks the hunter onto the nearest survivor. Win = anyone reaches the exit; lose = all cocooned or the 15-minute cap.
- **Hunter.** A `HunterController` seam (AI now, human-driven later) spawning a hostile Blighted-Kweebec role (a `Template_Kweebec_Razorleaf` Variant, `DefaultPlayerAttitude:Hostile`, bumped speed/ranges) with per-tick marked-target re-assert on Adventure-mode survivors.
- **Death -> cocoon.** A custom `OnDeathSystem` (ordered before the vanilla respawn screen) holds a downed in-round player dead-in-place, rescuable by a teammate; players outside a round are untouched. Revive runs `DeathComponent.respawn` at the instance spawn point.
- **Atmosphere + feedback.** Frozen dark midnight + forced Void weather; a custom round HUD (timer / objective / shrines / alive / corruption) with the native HUD stripped; `EventTitle` banners + Danger toasts; a server-pulsed 3-tier proximity heartbeat; a `ServerCameraService` top-down recipe for the future survival mode.
- **Integration is outbound native events only, zero MMO dependency.** `RoundStarted` / `RoundCompleted` / `PlayerCocooned` / `PlayerRescued` `IEvent<Void>` POJOs fired via `HytaleServer.get().getEventBus().dispatchFor(...)` behind a `hasListener()` guard.
- **Arena pack assets.** `instance.bson` (loaded as JSON) with `HytaleGenerator` + a vanilla flat WorldStructure; grove prefabs pasted at runtime via `PrefabUtil.paste`; the Blighted-Kweebec role; an en-US `.lang` (complete) and the round `.ui`.
- **Entry:** `/kweebec [start|exit|endall] [preset]` (the void-rift pad / shrine block / guide NPC triggers are designed-for, this is the first).
- **First playtest fixes.** The first live run surfaced three bugs, now fixed: (1) players spawned ~15 blocks underground because the spawn / arena Y did not match the `Default_Flat` worldgen floor (`Base` = 80) - `instance.bson` `SpawnPoint.Y` + `ArenaLayout` now sit at Y=80; (2) the round re-forced survivors to Adventure every tick, so an admin could not switch to Creative (it snapped back in ~1s) - game mode is now normalized ONCE on entry (`ChaseMode`, via a `PlayerRoundState.gameModeApplied` flag) and the hunter no longer touches it; (3) the HUD / title / toast strings rendered as raw keys because `kweebecnightmare.lang` keys carried the `kweebecnightmare.` prefix that `I18nModule` ALSO derives from the filename (doubling the namespace) - the file keys are now unprefixed.

Technical: the build targets the **live runtime** `HytaleServer.jar` (the `D:/Games/Hytale` 0.5.5 install the server actually launches), which matches the 0.5.3 decompile (org.joml math vectors, `DeathComponent.respawn`, the 4-arg `resetPlayer`, `HudManager.addCustomHud`, `exitInstance` returning a future). An early build mistakenly compiled against a stale secondary install (an AppData 0.5.3 jar from January) and produced a runtime `NoSuchMethodError`; re-pointing `gradle.properties` at the runtime jar fixed it. Every signature was javap-verified against the runtime jar. See the parent repo note `kweebec-nightmare-api-ledger`.
